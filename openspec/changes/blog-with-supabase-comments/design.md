## Context

PascubeDB é 100% estático (HTML/CSS/JS vanilla, sem bundler, sem backend). Dados vêm de Google Sheets + CSV fallback. Deploy via GitHub Pages.

Queremos adicionar blog com comentários sem quebrar a natureza estática do core. A solução é Supabase como backend externo — o frontend faz chamadas diretas via SDK.

## Goals / Non-Goals

**Goals:**
- Botão "Blog" no header ao lado de "Leaderboard"
- Modal `<dialog>` com lista de posts (fetch Supabase)
- Post expandido com markdown renderizado (via `marked`)
- Comentários por post com login GitHub OAuth
- RLS protegendo dados (público lê, autenticado comenta)
- Posts gerenciados via Supabase dashboard (admin)

**Non-Goals:**
- Editor de posts dentro do app (admin faz no dashboard)
- Notificações em tempo real (Supabase Realtime é futura melhoria)
- Sistema de roles ou permissões complexas (só admin vs autenticado)
- Moderação de comentários dentro do app (modera via dashboard)
- Nenhuma alteração nas specs existentes

## Decisions

**1. Supabase como backend**

Alternativas consideradas:
- GitHub Issues API + Device Flow OAuth: UX de login estranha pra web, dependência de rate limit anônimo
- Google Sheets + Apps Script: sem login real, latência de propagação, sem markdown

Supabase oferece PostgreSQL, auth embutido com GitHub OAuth, RLS, SDK CDN — tudo que precisamos sem backend próprio.

**2. Marked para renderizar markdown**

Alternativa: DOMPurify + marked (safe). Como os posts são escritos pelo admin no dashboard, risco XSS é mínimo. Ainda assim usar `DOMPurify.sanitize()` combinado com `marked.parse()`.

**3. Supabase Client via CDN (IIFE build)**

`@supabase/supabase-js` via jsdelivr como script tag. Sem ESM, sem bundler — consistente com o resto do projeto que usa CDN (Chart.js, Lucide).

**4. user_name desnormalizado na tabela comments**

Evita JOIN com `auth.users` a cada render. O nome vindo do GitHub OAuth é gravado direto na linha do comentário.

**5. Modal único com navegação interna (lista → post)**

Não cria múltiplos modais. Um único `<dialog>` que alterna entre view "list" e "post". Mais leve, menos DOM.

## Flows

```
┌──────────────────────────────────────────────────────┐
│                  BLOG FLOW                           │
├──────────────────────────────────────────────────────┤
│                                                      │
│  1. User clicks [Blog]                               │
│     → modal.showModal()                              │
│     → supabase.from('posts').select('*')             │
│     → render post list (title, date, comment count)  │
│                                                      │
│  2. User clicks a post                               │
│     → switch to post view                            │
│     → marked.parse(post.content)                     │
│     → supabase.from('comments').select('*')          │
│        .eq('post_id', id).order('created_at')        │
│     → render comments                                │
│                                                      │
│  3. User types comment + clicks Submit               │
│     → check session (supabase.auth.getSession())     │
│     → if no session: popup GitHub login              │
│        supabase.auth.signInWithOAuth({provider:'github'})│
│     → if session: INSERT comment                     │
│        supabase.from('comments').insert({...})        │
│     → append comment to list                         │
│                                                      │
│  4. User clicks "Back to list"                       │
│     → switch to list view                            │
│                                                      │
│  5. User clicks [X] or outside modal                 │
│     → modal.close()                                  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

## Data Model

```sql
-- Posts (admin escreve no dashboard)
create table posts (
  id bigint generated by default as identity,
  title text not null,
  content text not null,           -- markdown body
  slug text unique not null,       -- URL-friendly, tipo "mesa-26-2-release"
  published boolean default true,
  created_at timestamptz default now(),
  updated_at timestamptz default now(),
  primary key (id)
);

alter table posts enable row level security;

create policy "Posts public read"
  on posts for select
  using (published = true);

-- Comments (usuários logados)
create table comments (
  id bigint generated by default as identity,
  post_id bigint references posts(id) on delete cascade,
  user_id uuid references auth.users(id),
  user_name text not null,         -- desnormalizado do GitHub profile
  content text not null,
  created_at timestamptz default now(),
  primary key (id)
);

alter table comments enable row level security;

create policy "Comments public read"
  on comments for select
  using (true);

create policy "Comments auth insert"
  on comments for insert
  with check (
    auth.role() = 'authenticated'
    and user_id = auth.uid()
  );
```

## Component States

### Blog Button
- **Default**: ícone + "Blog" no header
- **Disabled**: não há (sempre disponível)

### Blog Modal — Post List
- **Loading**: skeleton cards (3x shimmer)
- **Empty**: "No posts yet. Stay tuned!"
- **Error**: "Failed to load posts. Try again." + retry button
- **Data**: cards com título, data formatada, nº comentários, seta

### Blog Modal — Post View
- **Loading post**: skeleton (title + body lines)
- **Loading comments**: small spinner abaixo do post
- **Post rendered**: markdown formatado, data, back button
- **Error post**: "Could not load this post"
- **Error comments**: "Could not load comments" + retry

### Comments Section
- **No comments**: "Be the first to comment!" + login or form
- **Has comments**: lista de comentários (avatar? nome, data, texto)
- **Not logged in**: "Login with GitHub to comment" button
- **Logged in**: textarea + Submit button
- **Submitting**: button disabled, "Posting..."
- **Submit success**: comment appended, textarea cleared
- **Submit error**: "Failed to post comment" + retry

## Risks / Trade-offs

| Risco | Mitigação |
|-------|-----------|
| Supabase free tier: 500MB DB, 50K monthly active users | Posts + comments desse projeto consomem < 1MB. 50K MAU é distante pra um dashboard de benchmark. |
| Supabase anon key exposta no client-side | É o modelo padrão do Supabase (público). RLS é a segurança. |
| GitHub OAuth redirect após login | Configurar redirect URL correta no Supabase dashboard (GitHub Pages URL). Fallback: testar em staging. |
| XSS via markdown posts | Usar DOMPurify.sanitize() em cima do output de marked.parse(). |
| Usuário fecha modal durante submit | Tratar com try/catch e não travar UI. Se o INSERT falhou por cancelamento, não tem efeito colateral. |
| marked + DOMPurify aumentam payload | Ambos via CDN (~30KB gzipped juntos). Aceitável. |
